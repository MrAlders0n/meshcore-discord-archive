Bubba_Nel 2025-12-04 2:53 p.m.
I wrote this in the main channel. Adding it here as it maybe helpful. Here's a couple things I can add..... This applies to using Li-Ion batteries. Other battery chemistries have simular issues. Most of the BMS boards out there have such a small hysteresis (the voltage above the undervoltage lock out, when the board reconnects the battery to the circuit) of only 1 - 2%. Depending on which BMS one uses, this amounts to about 0.02V which is not enough gain in voltage from the cutoff voltage; to save the battery from a voltage drop when the node begins to transmit. A better value would be like 10%; but they are not out there in the real world unless you design a circuit that would do that. Also, many of the BMS boards use a DW01A IC for the controller. This has a cutoff voltage of 2.5V which is very low, too low, and will cause damage to the Li-Ion battery if kept or recycled to this point for very long. At 2.5V the battery has virtually no capacity and will take time getting back to a good working level of 3.0V (if it even recovers at all). One should not be using batteries w/o some form of UVLO (under voltage lock out) BMS. UVLO is the voltage at which the circuitry cuts power from the battery to the load. This is what I use on all my batteries. These boards use the DW01A IC and I've purchase this IC from LCSC. Bought them in '23 for qty=50 for about $14 with shipping. This IC has a UVLO of 3.0V and a hysteresis of 0.2V Not the 10% that I would like, but the best out there. https://www.amazon.com/KOOBOOK-Protection-Solder-Lithium-Battery/dp/B07W75BQWW/ref=sr_1_1 https://www.lcsc.com/product-detail/C168774.html
entr0p1 2025-12-05 7:17 a.m.
@mrdan1023 I'm working on a branch of my own for this but it is a drastic change that the devs may not have the appetite to merge. We will see. But basically I've covered your scenario with a few different gates to prevent the brownout. They are: 1. Startup lockout - if voltage reading is < <n>mV, dont even boot. 2. Power states - 3 different load shedding stages based on several consecutive voltage readings below a threshold. When voltage returns above each threshold (again, consecutive readings) it will either step up to the next level or return to "Normal" mode. From lowest to highest severity: 2.a) Conserve mode - Disable repeats, don't reply to discover packets, disable guest login, disable sensors and GPS (if fitted) 2.b) Sleep mode - nRF deep sleep with a wake timer every <n> minutes to re-check voltage. 2.c) Shutdown mode - We're in the danger zone, power off the board. 3. POF Supervisor killswitch - If all the power states miss for whatever reason, the POF supervisor in the nRF can send an interrupt when the 3V3 rail is so low on voltage its about to collapse. The threshold is not configurable to my knowledge but it is low. We listen for that, and if received, go straight to the shutdown mode above. There is a "gate" to prevent any of this kicking in if USB or 5V power is applied to the board (i.e. we're dead/low, but charging).

Bubba_Nel 2025-12-05 1:19 p.m.
If one is going to be making s/w startup/lockout decisions based on the A/D readings from the battery port of a node; there are issues with that. I'm not saying it cannot be done but one needs to look at all the variables. Sorry if you already know this.... 1) Most Solar nodes are not using a battery connected to the battery connection on the node (RAK, XAIO, or nRF modules, V3, V4, etc.) as they have found that the onboard charger is not appropriate for longterm battery charging capabilities, especially in winter. That is typically where the battery's voltage is measured. So they are using external chargers which are typically connected to the node through their +5V inputs. The A/D circuitry is not in play. It seems many of the portable, handheld nodes ARE using Li-Ion cells connected through the battery connection, so current s/w in this case would work. 2) Many nodes are using batteries other than Li-Ion, which has been the standard for a long time. Other chemistries can be LiFePO4, LTO, Silicon-Ion, and even capacitor. Each of these have different nominal voltage values, and much different discharge curves. All of which cannot be connected to the battery connection without some h/w manipulation (i.e. a buck/boost converter). These converters supply a constant voltage and therefore s/w cannot properly read it. If one would want to measure battery conditions in this case, one could do it with something like a INA219, or INA3221. Then s/w could make some decisions on battery status. This means the s/w would need now to read the I2C port and make decisions from the readings.

First of all, let me say I really appreciate the effort and thought that went into this reply. This is awesome discourse and helps make sure the right levers are being pulled. RE your concerns, in order: 1. You're correct, and in the cases where folks are using external charging circuitry my (maybe flawed) assumption would be that all of the safety cutoffs should be handled within that external board. The fixes I'm building are purely for those of us using the built-in charging and battery connectors. It also keeps things simple for voltage thresholds as we know what chemistry each board natively supports. Just for completeness sake, I'll mention the change I'm working on is only for nRF52840 boards. Others will be a totally different architecture and require careful thinking of their own. Within the nRF52 architecture, each board needs to be manually flagged as "supported" for this power management code as not all may have the needed components, and all of the power management (except startup cutoff) can be disabled in the node preferences (by default they are turned off, in fact). Also, if power is applied to the 5V input or USB port, even with a battery connected, all of the power management stuff is effectively in bypass. This is to stop the node flapping when charging or when connected to less than ideal USB power (e.g. a long cable). 2. Valid point, but as far as I'm aware the native charging mechanism on the boards is Li-Ion/Li-Po (I.e. 3.7V) in every case that I'm aware of. Still, that doesn't mean these other chemistries should be excluded from some of the power management as it would be nice to not go straight from normal operation to total power cut and be able to potentially keep it alive long enough for the sun to return with the right load shedding. Its good consideration for what the next stage might look like if this pans out.
