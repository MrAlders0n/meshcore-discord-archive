from ai: Hypothesis A (most common): MeshoS is missing the equivalent of INPUT_PULLUP If MeshoS configures the pin as plain input (floating) or uses the wrong pull direction, you’ll get “some devices work, some don’t,” depending on leakage and tolerances. Hypothesis B: MeshoS uses the wrong interrupt edge (RISING vs FALLING) Meshtastic forces FALLING for T-Deck. If MeshoS uses RISING, it may miss the press depending on wiring. Hypothesis C: MeshoS uses a different threshold/debounce than Meshtastic Meshtastic sets TB_THRESHOLD 3. If MeshoS requires more (or does time-based debounce badly), marginal devices might never hit the threshold on DOWN specifically (direction switches can differ mechanically). Hypothesis D: GPIO15-specific nuance on some boards On ESP32-class chips, some GPIOs have bootstrapping or special behavior. If MeshoS configures GPIO15 differently (drive strength, hold, pulldown, etc.), it might interfere on some devices.
