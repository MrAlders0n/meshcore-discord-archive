Delltar 2025-10-22 7:40 a.m.
This is a bit of a wild idea maybe, but I noticed that in some locations (like transmitter towers or particularly located buildings) people have to use multiple repeaters to prevent one side from being in a shadow. Situation where there must be two repeaters on a single transmitter tower probably isn't ideal and so I wanted to float the (maybe a bit ambitious) idea of possibly connecting multiple repeater nodes into one logical repeater. Now the reality is probably complicated for this, I am not aware of any existing LoRa hardware that would have multiple LoRa modules to accommodate multiple antennas. It might also not be practical to actually have such HW since in reality the "multi antenna repeater" would need to have some distance between the antennas and leading all of them to one box is probably not efficient. Connecting multiple physical nodes to create one logical repeater might also be a tall order, especially if those nodes run from solar where maintaining active wifi connection, or running some sort of central devices to coordinate everything might not be practical. Though maybe connecting multiple nodes to raspberry pi via BLE and powering the pi from solar is not so far fetched. So as you can see, this is more idle musing from me rather than some concrete idea, but wanted to float it anyway

Scott 2025-10-23 12:25 a.m.
Yeah, the transports codes are a filtering mechanism which repeaters will interrogate. They will be for 2 purposes, 1) for limiting flood broadcasts to just a subset (ie. sub-mesh) of repeaters, 2) enabling completely private repeater infra. The codes will go thru an algo that takes a Transport Key + packet info -> transport code. For private infra the tranport keys are only disclosed to participating nodes. For 1) these will be like the #channel names, where the Transport Key is auto-generated, ie. public knowledge.
