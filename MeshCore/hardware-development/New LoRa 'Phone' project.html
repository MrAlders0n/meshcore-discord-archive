Mike WB6KTN 2026-03-24 2:48 p.m.
You will likely need to implement a protocol on top of the physical layer with additional FEC to support streaming audio over an unreliable channel with no feedback. You will likely need to update the channel loss model to factor in LoRa's chirp modulation and everything LoRa already does to spread the FEC symbols across the packet to mitigate burst loss. It is a non-trivial problem. Loss in a low bit rate codec is devistating. Codecs that can deliver understandable voice in the presence of loss can only do so by having unneeded information that can be lost without impacting understandibility. There is nothing special about these chips. Give them a packet, wait for the interrupt. Wait for a packet and add it to the queue. The receive jitter buffer takes some effort but that is independent of the device. background: implemented a dvb/mbms streaming protocol in a former life.

I didnt try it. It can work in theory but if you do that you flood the airways and can cause issues. For example if both are transmitting at the same time, neither will get it because no one is in RX mode, so you would pretty much need to stop a user from sending voice data until no one is sending voice data.
