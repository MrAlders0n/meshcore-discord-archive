# Pager Mode Implementation Notes ## Overview - Pager messages remain standard room text (`PAYLOAD_TYPE_TXT`); they are human-readable and display normally on non-pager clients. - Dispatch rooms advertise capability (TLV bit) so non-pager clients ignore the flag; pager-aware clients switch behavior. - ACKs: reuse existing two-shot ACK flow (multipart pre-ACK + final ACK); return paths enable direct retries after initial flood. ## Capability Signaling - Room servers mark pager capability via advert `feat1` bit (`ADV_FEAT1_DISPATCH` in `extra1`). Unknown flags are ignored by non-pager clients; pager-aware clients switch behavior when the bit is set. Payload type/version stays unchanged. - Dispatch room names should also be prefixed with `[DISPATCH]` as a human hint; pager software ignores the prefix for logic but users immediately see they joined a dispatch room. ## Message Format - Standard room text payload, no special type. Keep human-readable tagging, e.g. `1/1 [GRP:A,C] [TON:A] EMS 2030 Benson Road, Alpha lift assist. Respond Fire 2`. - Non-pager clients render the message verbatim; pager clients may parse tags for UI/alerting but must leave payload interoperable. ## Preferences & Migration - Core prefs blob (from companion firmware) stays intact; pager prefs are additive (pager_enabled off by default, two-shot ACK on, heartbeat/slots defaults). - Migration strategy: load existing prefs, preserve all core fields (freq/BW/SF/CR/tx power, names, telemetry flags), then apply pager defaults when enabled. No change to payload versions.

## Pager Operational Rules - Pager nodes are inert until connected to a dispatch room (capability bit set). Attempts to send messages when disconnected should fail loudly (UI alert: buzzer/LED) and not transmit. - Telemetry is pull-only: dispatch issues a request; pagers reply in order using a slotted window to avoid collisions. Pagers never auto-send telemetry. - Two-shot ACKs with optional slotting (by pager ID) for pages and telemetry replies to reduce collisions on busy meshes. - Pager nodes cache the last joined dispatch identity in a pager-only file (`/pager_dispatch.bin`, versioned) so they can reconnect and enforce “one dispatch” without touching existing prefs/contacts. - Alert severities (pager mode): - A: soft chirp (single). - B: double chirp. - C: short rising triple. - D: urgent triple. - E: latched high-priority (keeps repeating until acknowledged via button). RTTTL patterns live in `examples/companion_radio/PagerAlertTones.h`; E is latched by default. - Default pager behavior (no screen): dispatch text messages trigger alert D; `[PRIO:X]` tag (A–E) in the body overrides the level; button press stops any latched alert. Channel traffic from dispatch follows the same rule. - Disconnect handling: pager nodes watch the cached dispatch connection; if disconnected for >10s, raise alert C and blink LED (if available) until a connection is restored or the user acks. - Slotting defaults (host-only helper for planning): 8 slots over 2400 ms for ACK/telemetry replies, derived from pager ID hash; two-shot ACK stays enabled by default. See `test/lib/pager_prefs/SlotPlanner.h` and `test/test_pager_slots` for deterministic slot planning.

## Tests (host, PlatformIO native_test) - `pio test -e native_test` runs host-only Unity suites: - `test_core_prefs`: serialize/deserialize current prefs blob + lat/lon (simulates upgrade from existing firmware). - `test_pager_prefs`: pager prefs round-trip. - `test_pager_migration`: upgrade helper preserves core values, sets pager defaults. - `test_pager_format`: parses bell-prefixed pager text, ignores non-pager text. - `test_ack_hash`: verifies ACK hash helper is stable across identical pager messages and differs when content changes. - Helpers are guarded with `UNIT_TEST` to avoid firmware impact; upstream code paths unchanged.
