@CSP-Tom and I have been working on a wardriving app with multi-region support: https://meshmapper.net/ Requirement: you’ll need an existing MeshCore Packet Analyzer Observer running. Huge thanks to @🌲 Tree for making this possible. How it works
  • Frontend: a BLE WebApp that performs the wardriving (originally a fork)
  • Backend: stores/organizes the data + collects packets from MeshCore Packet Analyzer Demo (Ottawa / YOW — currently the most populated)
Map: https://yow.meshmapper.net/ Leaderboard: https://yow.meshmapper.net/leaderboard.php Try this:
  • Click on a repeater
  • Explore the layers
    • Noise Heatmap is new
  • Check the leaderboards
  • Click on various coverage blocks Features
  • Administration controls: limit wardrive sessions, disable wardriving entirely, or allow RX-only (passive) wardriving per region
  • Collect noise floor data and map city-wide noise
  • Graph multiple types of coverage zones
  • Graph repeater coverage
  • Track how far repeaters can reach
  • Plus a bunch of other things I’m probably forgetting Repeaters + duplicates (important)
  • If your region has unique repeater IDs, you’ll get some very cool repeater visualizations/stats.
  • If your region has duplicate repeater IDs, the app still works, but it will exclude pings from repeater stats for IDs that are known duplicates (to keep the data sane). Want your region onboarded?
Head here https://meshmapper.net/?onboarding and fill out the web form! Want to request a feature/report and issue. https://github.com/MeshMapper/MeshMapper_Project (edited)

CSP-Tom 2026-01-23 5:23 p.m.
A little about how duplicate repeaters work with MeshMapper... When a repeater adverts and it's sharing the same 2-digit ID as another, both that repeater and the existing one will enter an "excluded" state. This prevents any calculation of max distance because there is no way to tell which repeater was involved in a ping. New data coming in with that repeater ID involved (it heard a repeat from that ID, that ID was involved as a hop, etc.) will forever be flagged as not knowing exactly where the repeater was, and will never be involved in any max distance or "line drawing" calculations. Duplicate repeaters are checked automatically every day and if one of those repeaters hasn't sent an advert in 3 days, it's removed from the database and the other conflicting repeater (as long as it HAS sent an advert recently) is fully reactivated. You can speed this process up if you're a region administrator, as the admin portal will flag for you any duplicates and allow you to edit accordingly (for instance if you're aware that the duplicate ID was temporary and has since taken on a new ID). MeshMapper Session Management When you connect to your MeshCore device, MeshMapper acquires a session slot from the regional API. Sessions are renewed automatically whenever you submit data to the API (TX ping results, RX packets, discovery responses). This means active wardriving keeps your session alive naturally. For Active Mode and Passive Mode, there's an additional safeguard: a heartbeat mechanism that renews your session one minute before expiration if no API submissions have occurred. This ensures continuous wardriving sessions don't unexpectedly drop. However, if you're just connected and idle (or only send a single manual ping), the session will expire after the timeout period (~5 minutes from your last API submission). The next action you attempt - starting Active/Passive Mode or sending another ping - will trigger a "session invalid" error, requiring reconnection. This is intentional. Session slots are a limited resource used to enforce slot limits that protect the mesh network from overload. If you're not actively wardriving, MeshMapper releases your slot so other wardrivers in the region can use it. Idle connections don't get to hold slots indefinitely.
